## CDK Context Values and Parameters

In AWS CDK for Python,Â **parametersÂ (AWS CloudFormation Parameters) are used for values supplied at deployment time, whileÂ context valuesÂ are primarily for build-time configuration and lookups from the AWS environment**. Context values are the recommended way to pass information into a CDK app at synthesis time.

**Parameters (CloudFormation Parameters)**

Parameters are a native CloudFormation feature and are specified at deployment time. The CDK team generally recommends using context values or environment variables instead, as parameters cannot be easily used for control flow in your CDK code.

**Defining a Parameter in Python**

You define a parameter using theÂ `CfnParameter`Â class:

```python
import aws_cdk as cdk
from aws_cdk import aws_cloudformation as cfn

class MyStack(cdk.Stack):
    def __init__(self, scope: cdk.App, id: str, **kwargs) -> None:
        super().__init__(scope, id, **kwargs)

        # Define a CloudFormation parameter
        my_param = cfn.CfnParameter(self, "MyParameter",
            type="String",
            default="default-value",
            description="A custom parameter for the stack"
        )

        # Use the parameter value
        # This value is a CloudFormation intrinsic function (a Token) and not available at synthesis time
        param_value = my_param.value_as_string
```

**Providing Parameter Values**

- **During deployment:**Â Use theÂ `-parameters`Â option with theÂ `cdk deploy`Â command:

    **bash**

    `cdk deploy --parameters MyParameter=new-value`

- **In the AWS CloudFormation console:**Â You will be prompted for values if you deploy manually through the console.

**Context Values**

Context values are key-value pairs that are resolved atÂ **synthesis time**Â (when you runÂ `cdk synth`). They are ideal for environment-specific configurations (e.g., dev vs. prod instance sizes), caching environment data (like AMIs or VPC IDs), and for internal CDK operations like feature flags.

**Sources of Context Values**

Context values can come from several places, with command-line values taking precedence over file-based values:

- **CDK Toolkit (`cdk`Â command):**Â Using theÂ `-context`Â (`c`) option.
- **`cdk.json`:**Â Storing user-defined context values.
- **`cdk.context.json`:**Â Managed by the CDK CLI to cache values retrieved from your AWS account (e.g., Availability Zones, VPC lookups).Â **Do not edit this file manually.**
- **In your app:**Â Using theÂ `construct.node.set_context()`Â method.

**Using Context Values in Python**

You can retrieve context values within your Python application using theÂ `try_get_context`Â method:

```python

    # Handle the case where the context value is not provided
    print("Warning: 'configName' context value is missing, using default.")

```

**Providing Context Values**

- **Command Line:**Â Pass runtime values using theÂ `-context`Â flag:

    **bash**

- **`cdk.json`Â file:**Â Define values in your project'sÂ `cdk.json`Â under theÂ `context`Â key:

## CDK Parameters vs Context Values

### CDK Parameters (`CfnParameter`)

**What they are:**
- CloudFormation parameters that prompt for values at **deployment time**
- Defined in your CDK code but values provided when running `cdk deploy`
- Part of the CloudFormation template itself

**Characteristics:**
- âœ… Can change values between deployments without changing code
- âœ… Interactive prompts during deployment
- âœ… Good for values that differ per deployment
- âŒ Can't use in synthesis-time logic (if statements, loops)
- âŒ Limited to CloudFormation types (String, Number, List, etc.)
- âŒ Can't be used to conditionally create resources

**Use when:**
- Value changes frequently between deployments
- Different operators need different values
- You want deployment-time flexibility
- Building reusable CloudFormation templates

**Example:** Database password, instance sizes that operators choose

---

### CDK Context Values

**What they are:**
- Configuration values available at **synthesis time** (when running `cdk synth`)
- Can be provided via CLI, `cdk.json`, or `cdk.context.json`
- Become hardcoded in the synthesized CloudFormation template

**Characteristics:**
- âœ… Available during synthesis - can use in if/else logic
- âœ… Can conditionally create/skip resources
- âœ… Can be any data type (strings, objects, arrays)
- âœ… Cached in `cdk.context.json` for consistency
- âŒ Require re-synthesis to change values
- âŒ Not visible in CloudFormation console

**How to provide:**
```bash
# Via CLI
cdk deploy -c github_repo=owner/repo

# Via cdk.json
{
  "context": {
    "github_repo": "owner/repo"
  }
}
```

**Use when:**
- Value is known at development time
- Need to use value in synthesis logic
- Building environment-specific stacks
- Value is configuration, not secret

**Example:** Environment names, feature flags, account IDs, repo names

## Recommended Implementation Patterns

**Pattern 1: Context via cdk.json**
```python
github_repo = self.node.try_get_context("github_repo")
if not github_repo:
    raise ValueError("Context value 'github_repo' is required")
```

```json
// cdk.json
{
  "context": {
    "github_repo": "avr2002/files-api"
  }
}
```

**Pattern 2: Environment Variable**
```python
github_repo = os.environ.get("GITHUB_REPO")
if not github_repo:
    raise ValueError("GITHUB_REPO environment variable required")
```

**Pattern 3: CLI Context**
```bash
cdk deploy -c github_repo=avr2002/files-api
```

## When Parameters ARE Better

Use Parameters when you need:
- Operators to make runtime decisions (e.g., instance type selection)
- Different values for same template across accounts
- CloudFormation StackSets with different parameter values
- True infrastructure-as-a-template (not infrastructure-as-code)

Example: A reusable template deployed by multiple teams with their own values.
